# 整体院・整骨院 Legal Watcher 要件定義 v0.1
対象フェーズ：Phase 1〜Phase 4

## 0. プロジェクト概要

### 目的
整体院・整骨院・接骨院・鍼灸院・あん摩マッサージ指圧院に関係しうる法令、通知、行政資料、療養費関連情報、広告規制、行政処分、消費者庁・厚労省・e-Gov等の更新情報を定期監視し、日次・週次レポートとして自動生成する。

### 基本方針
- 情報取得・差分検知・URL管理はルールベースで行う。
- LLMには「調査」を丸投げしない。
- Cursor SDKは、取得済み原典の要約・分類・実務影響整理・レポート整形に使う。
- 原典URL、取得日時、差分、判断根拠を必ず保存する。
- 法的判断は断定せず、「要原典確認」「要専門家確認」「要自治体確認」を明示する。
- まずは公開情報のみを対象にし、患者情報・院内機密情報は扱わない。
- 将来的に機密情報を扱う場合は、Ollama等のローカルLLM連携を別系統として検討する。

### 想定ユーザー
- 整体院・整骨院・接骨院の運営者
- 院内の広告・LP・SNS・業務マニュアルを管理する担当者
- 法令・行政通知・療養費改定・広告規制の変更を継続的に追いたい人

## 1.5 機能要件

### F1-1：情報取得
- RSSを取得できること。
- HTMLページを取得できること。
- PDFリンクを検出できること。
- PDF本文を抽出できること。
- e-Gov API等のAPIレスポンスを取得できること。
- 取得時刻、HTTPステータス、URL、タイトル、本文抜粋を保存すること。

### F1-2：差分検知
- URL単位で前回取得時のハッシュを保存すること。
- タイトル、本文、PDF本文、リンク一覧の差分を検知できること。
- 新規追加、更新、削除、取得失敗を区別すること。
- 差分がないものはCursor SDKに送らないこと。

### F1-3：一次フィルタ
- 監視キーワードに一致する更新を優先抽出すること。
- 公式ソース種別に応じて重み付けできること。
- 例：
  - 厚労省の療養費関連更新：高優先
  - 消費者庁の措置命令：高優先
  - 自治体の施術所関連更新：中〜高優先
  - 一般的な健康ニュース：低優先

### F1-4：Cursor SDK連携
- 差分検知された更新情報をCursor SDKへ送信すること。
- LLMへの入力には以下を含めること。
  - タイトル
  - 情報源名
  - 原典URL
  - 公開日
  - 取得日
  - 差分
  - 本文抜粋
  - PDF本文抜粋
  - 一次フィルタ結果
- LLMには以下を出力させること。
  - 重要度
  - 対象業態
  - カテゴリ
  - 要約
  - 実務影響
  - 広告・LP・SNSへの影響
  - 確認ポイント（`operator_checkpoints`。表示ラベルは `display.yaml` の `operator_label` で設定可）
  - 専門家確認の必要性
  - 原典URL
  - 信頼度
  - 不明点

### F1-5：日次レポート生成
- `reports/daily/YYYY-MM-DD.md` を生成すること。
- 更新がない日も「更新なし」レポートを生成できること。
- 重要度順に並べること。
- 原典URLを必ず表示すること。
- LLMの断定を避け、「可能性」「要確認」を明示すること。

### F1-6：GitHub Actions定期実行
- 毎日JST 08:00に実行すること。
- 手動実行 `workflow_dispatch` に対応すること。
- 生成されたレポートと状態ファイルをコミットすること。
- エラー発生時はGitHub Actions上で確認できること。

## 1.6 非機能要件

### 信頼性
- 取得失敗しても全体処理を止めない。
- ソース単位でエラーを記録する。
- リトライを行う。
- 同一URLの重複登録を避ける。

### 監査性
- 原典URLを保存する。
- 取得日時を保存する。
- 差分テキストを保存する。
- LLM要約と原文を分けて保存する。

### コスト管理
- 差分がないものはLLMに送らない。
- PDF全文を丸ごと送らず、差分・該当箇所・抜粋を送る。
- 日次は短め、週次は詳細めにする。

### セキュリティ
- APIキーはGitHub Secretsに保存する。
- public repositoryでは運用しない。
- 患者情報や院内機密情報はPhase 1では扱わない。

# Phase 2：MarkdownレポートをObsidian Vaultにも同期

## 2.1 目的

生成された日次・週次レポートをObsidianで蓄積・検索・タグ管理できるようにする。

## 2.2 追加機能要件

### F2-1：Obsidian出力

* Obsidian Vaultのパスを設定できること。
* レポートをObsidian Vaultへコピーできること。
* 出力先構成は以下とする。

```text
Obsidian Vault/
└─ Legal Watch/
   ├─ daily/
   ├─ weekly/
   ├─ topics/
   │  ├─ 柔整療養費/
   │  ├─ あはき療養費/
   │  ├─ 広告規制/
   │  ├─ 景品表示法/
   │  ├─ 医業類似行為/
   │  ├─ 行政処分/
   │  └─ 自治体通知/
   └─ sources/
      ├─ 厚労省.md
      ├─ 消費者庁.md
      ├─ e-Gov.md
      ├─ 官報.md
      └─ 自治体.md
```

### F2-2：Frontmatter

* 各MarkdownにYAML Frontmatterを付与すること。

```yaml
---
type: legal-watch-daily
date: 2026-05-25
source_count: 12
high_priority_count: 1
medium_priority_count: 3
low_priority_count: 5
tags:
  - legal-watch
  - 整体院
  - 整骨院
  - 法令監視
---
```

### F2-3：タグ付け

* カテゴリごとにObsidianタグを付与すること。
* 例：

  * `#柔整療養費`
  * `#広告規制`
  * `#景表法`
  * `#厚労省`
  * `#消費者庁`
  * `#自治体通知`
  * `#要専門家確認`

### F2-4：関連ノート生成

* 高重要度の項目は、日次レポートとは別にトピック別ノートにも追記できること。
* 例：

  * `topics/柔整療養費/2026-05-25_厚労省資料更新.md`
  * `topics/広告規制/2026-05-25_消費者庁措置命令.md`

### F2-5：インデックス生成

* `Legal Watch/index.md` を生成・更新すること。
* 最新レポート、重要度高の項目、要確認項目へのリンクを表示すること。

## 2.3 同期方式

### GitHub Actions運用の場合

* GitHub Actionsで生成したMarkdownをGitHub repoに保存。
* Obsidian Vaultが別環境にある場合は、ローカル同期スクリプトでpullする。
* GitHub Actionsから直接ローカルObsidianへは書き込まない。

### Mac mini運用の場合

* Mac mini上で日次バッチを実行。
* レポート生成後に直接Obsidian Vaultへ書き込む。
* 将来Tauriアプリ化する場合はこの方式を標準とする。

## 2.4 受け入れ条件

* Obsidian Vault内に日次レポートが保存される。
* Frontmatterとタグが正しく付与される。
* `index.md` から最新レポートへ移動できる。
* 重要度高の項目がトピック別に分離される。
* GitHub保存用レポートとObsidian用レポートの内容が矛盾しない。

# Phase 3：週次レポート、広告チェックリスト、院内マニュアル影響確認

## 3.1 目的

日次の更新情報を週次で整理し、実務に使える形へ変換する。
特に広告・LP・SNS・院内マニュアルへの影響を確認できるようにする。

## 3.2 追加機能要件

### F3-1：週次レポート生成

* 毎週月曜JST 08:30に前週分の週次レポートを生成すること。
* 出力先：

  * `reports/weekly/YYYY-WW_legal_watch.md`
  * `Obsidian Vault/Legal Watch/weekly/YYYY-WW_legal_watch.md`

### F3-2：週次レポート構成

週次レポートは以下の構成にする。

```md
# 整体院・整骨院 Legal Watch Weekly
対象期間：YYYY-MM-DD〜YYYY-MM-DD

## 1. 今週の重要更新
## 2. 整骨院・接骨院への影響
## 3. 整体院への影響
## 4. 鍼灸・あん摩マッサージ指圧への影響
## 5. 療養費・受領委任関連
## 6. 広告・LP・SNSへの影響
## 7. 消費者トラブル・健康被害関連
## 8. 自治体・保健所関連
## 9. 確認ポイント（Operator が確認すべきこと）
## 10. 専門家確認候補
## 11. 原典一覧
```

### F3-3：広告チェックリスト生成

* 関連更新がある場合、広告・LP・SNS用チェックリストを生成すること。
* 出力先：

  * `reports/checklists/YYYY-MM-DD_ad_checklist.md`
  * `Obsidian Vault/Legal Watch/checklists/YYYY-MM-DD_ad_checklist.md`

### F3-4：広告チェック項目

以下の観点を含めること。

```md
# 広告・LP・SNSチェックリスト

## 表現リスク
- 「治る」と断定していないか
- 「必ず改善」と保証していないか
- 医療機関と誤認される表現がないか
- 国家資格の有無を誤認させていないか
- ビフォーアフターが過度な効果保証に見えないか
- 口コミ表示に不自然な誘導がないか
- No.1表示の根拠が明確か
- 期間限定・割引表示に誤認がないか

## 業態別確認
- 整体院として使える表現か
- 整骨院として使える表現か
- 柔道整復師の業務範囲と矛盾しないか
- 鍼灸・あん摩マッサージ指圧の表示として妥当か

## 要確認
- 原典確認が必要な項目
- 自治体確認が必要な項目
- 顧問・専門家確認が必要な項目
```

### F3-5：院内マニュアル影響確認

* 更新情報から、院内マニュアルに影響しそうな項目を抽出すること。
* 例：

  * 受付説明
  * 施術メニュー表記
  * 問診票
  * 同意書
  * 療養費請求フロー
  * スタッフ説明文
  * LP掲載文
  * SNS投稿ルール
  * 返金・解約説明
  * 施術リスク説明

### F3-6：実務アクション抽出

* 各重要更新に対して、次のようなアクション項目を生成すること。

```md
## 実務アクション
- [ ] 原典PDFを確認する
- [ ] 施行日・適用日を確認する
- [ ] 対象業態を確認する
- [ ] 現在のLP表現に影響があるか確認する
- [ ] スタッフ説明文を見直す
- [ ] レセコン業者へ確認する
- [ ] 自治体窓口へ確認する
- [ ] 顧問・専門家へ確認する
```

### F3-7：転用文生成

* レポート内容を以下に転用できるようにする。

  * 院内共有メモ
  * スタッフ向け説明
  * SNS投稿の下書き
  * ブログ記事の下書き
  * 顧問に送る確認メール下書き

### F3-8：転用文の制約

* 法的断定をしない。
* 「当院は法律上問題ありません」等の保証表現を避ける。
* 一般読者向け投稿では不安を煽らない。
* 原典確認前の内容は「確認中」「可能性」と明記する。

## 3.3 週次Cursor SDKプロンプト要件

Cursor SDKには以下の役割を与える。

```text
あなたは整体院・整骨院・接骨院・鍼灸院向けの法定情報ウォッチャーです。

目的：
日次レポート群をもとに、前週の重要更新を整理し、
実務上の確認ポイント、広告表現への影響、院内マニュアルへの影響、
専門家確認が必要そうな論点を抽出してください。

制約：
- 法的判断を断定しない。
- 原典URLを必ず残す。
- 不明点は不明と書く。
- 「要原典確認」「要自治体確認」「要専門家確認」を明示する。
- 整体院、整骨院、接骨院、鍼灸院、あん摩マッサージ指圧院で対象を分ける。
- Operator が実務で確認しやすいチェックリスト形式にする。
```

## 3.4 受け入れ条件

* 週次レポートが自動生成される。
* 広告チェックリストが生成される。
* 院内マニュアル影響候補が抽出される。
* 転用文が生成される。
* 転用文に法的断定が含まれない。
* 要専門家確認項目が明示される。

# Phase 4：Tauriデスクトップアプリ化

## 4.1 目的

CLI / GitHub Actions運用から、日常的に確認しやすいデスクトップアプリへ拡張する。
Mac miniまたはMac上で常駐し、更新確認、レポート閲覧、監視先管理、Obsidian出力、手動実行をGUIで行えるようにする。

## 4.2 技術スタック

* Tauri v2
* TypeScript
* React または Svelte
* SQLite
* Rust backend
* 既存TypeScript監視ロジックは可能な限り再利用
* Cursor SDK連携はNode sidecarまたは外部バッチとして実行
* 将来的なOllama連携も追加可能な設計にする

## 4.3 アプリ形態

* デスクトップアプリ
* メニューバー常駐対応
* 手動実行ボタン
* 日次・週次レポート閲覧
* 通知機能
* 設定画面
* ログ画面

# 共通要件

## C1：重要度分類

### high

* 厚労省の療養費・受領委任・オンライン請求関連
* 法令改正・省令改正・通知改正
* 消費者庁の措置命令・課徴金
* 柔整・あはき・整体に直接関係する行政処分
* 広告規制に直接影響する情報
* 健康被害・重大事故・注意喚起

### medium

* 自治体の施術所関連ページ更新
* パブリックコメント開始
* 検討会資料更新
* 周辺法令の変更
* 広告・消費者契約・表示に関係しそうな情報

### low

* 一般的な健康情報
* 間接的な制度更新
* 参考資料更新
* 関係可能性はあるが実務影響が不明なもの

## C2：カテゴリ分類

* 柔整療養費
* あはき療養費
* 受領委任
* オンライン請求
* 広告規制
* 景品表示法
* 医業類似行為
* 健康被害
* 行政処分
* 自治体通知
* パブリックコメント
* 法令改正
* 消費者トラブル
* その他

## C3：対象業態分類

* 整体院
* 整骨院
* 接骨院
* 柔道整復師
* 鍼灸院
* あん摩マッサージ指圧

```json

  "importance": "high | medium | low",
  "category": "string",
  "target_business": ["整体院", "整骨院"],
  "summary": "string",
  "what_changed": "string",
  "impact": "string",
  "ad_impact": "string",
  "action_items": [
    "string"
  ],
  "needs_original_check": true,
  "needs_local_government_check": false,
  "needs_expert_review": true,
  "confidence": 0.0,
  "unknowns": [
    "string"
  ],
  "source_url": "string"
}
```

## C5：レポート文体

* BOSSがすぐ判断できる実務寄りの文体にする。
* 法的断定は避ける。
* 「何が変わったか」「何に影響するか」「何を確認するか」を明確にする。
* 長文の法律解説より、確認ポイントと原典導線を重視する。
* 重要度高の項目は冒頭に置く。
* 原典URLは必ず残す。

## C6：禁止事項

* LLMにWeb調査を丸投げしない。
* 原典URLのない情報をレポートの主根拠にしない。
* 「合法」「違法」と断定しない。
* 患者情報をCursor SDKへ送らない。
* 院内機密資料をCursor SDKへ送らない。
* SNS転用文で不安を煽らない。
* 広告転用文で効果保証しない。
