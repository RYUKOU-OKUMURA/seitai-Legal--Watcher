# 整体院・整骨院 Legal Watcher 技術スタック v0.1
対象：Phase 1〜Phase 4

## 0. 全体方針

### 基本アーキテクチャ

公開情報の取得・差分検知は自前コードで堅く実装する。
Cursor SDKは、取得済み情報の要約・分類・レポート整形に使う。

構成：

  公式情報源
    ↓
  TypeScript Fetcher
    ↓
  差分検知 / スコアリング
    ↓
  Cursor SDK
    ↓
  Markdownレポート
    ↓
  GitHub / Obsidian / Tauri App

---

### 言語

- TypeScript
  - Phase 1〜3のメイン実装

- Rust
  - Phase 4のTauriバックエンド
  - ローカルファイル操作、SQLite、通知、OS連携を担当

- SQL

- Markdown

  - Cursor用プロンプト/Skill定義

## 17. Phase 4でのCursor SDK実行方式

### 推奨方式

Tauriアプリ本体：

  React UI
    ↓
  Tauri Rust backend
    ↓
  Node.js sidecar
    ↓
  Cursor SDK

理由：

- Cursor SDKはTypeScript/Node側に置く方が自然
- 既存Phase 1〜3のコードを再利用できる
- RustへLLM連携を移植しなくてよい
- GitHub Actions版とローカルアプリ版で共通ロジックを使える

### 非推奨

- Cursor SDK処理をフロントエンドブラウザ側に置く
- API KeyをReact側に保持する
- TauriアプリからLLMに丸投げしてWeb調査させる

# 28. 設計上の重要判断

## Cursor SDKはバックエンド側で使う

ブラウザUI側では使わない。

理由：

- API Keyを守るため
- GitHub ActionsでもTauriでも共通化するため
- LLM処理をテストしやすくするため

## LLMにはWeb調査をさせない

LLMには、すでに取得した原典情報だけを読ませる。

理由：

- 見落とし防止
- 原典URLの保持
- 監査性
- 再現性
- コスト管理

## Phase 4でもTypeScript資産を捨てない

Tauri化しても、Phase 1〜3のTypeScriptバッチはNode sidecarとして再利用する。

理由：

- 開発速度が速い
- Cursor SDK連携をそのまま使える
- GitHub Actions版とデスクトップ版を同時に維持できる

## SQLite移行は早めにやる

Phase 1はJSONでよいが、Phase 3〜4ではSQLiteが望ましい。

理由：

- 更新履歴検索
- ステータス管理
- レポート再生成
- Tauri GUIとの相性
- 監査ログ管理
